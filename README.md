# OCR Statement-of-Loss Extraction

A small, runnable document-intelligence pipeline that ingests a **Statement of
Loss** PDF, OCRs it, parses line items, validates them against a JSON schema,
and routes low-confidence rows to a human review queue before emitting
structured JSON.

```
PDF  ->  OCR  ->  parse line items  ->  schema validation  ->  confidence router  ->  JSON
                                              |                        |
                                          (reject)              (review_queue/)
```

> ### ⚠️ Synthetic data only
> **Every byte of data in this repository is synthetic and generated by code.**
> There are no real claims, claimants, policies, dollar amounts, or PII anywhere
> in this project. The sample PDF is produced by `generate_pdf.py` from
> hand-authored fake rows in `sample_data.py`. This is a self-contained
> implementation that demonstrates the extraction-and-routing pattern; it is not
> connected to any production system or real dataset.

---

## Why this exists

Most extraction demos stop at "PDF in, JSON out" on clean documents. In
practice, document pipelines are dominated by the hard cases: a smudged total, a
row that straddles a page break, a digit that OCR transposes with full
confidence. This project is built around that reality. The interesting code is
not the happy path; it is the **confidence-threshold router** and the
**defenses against specific OCR failure modes**, both of which are documented
and tested below.

### Two gates, on purpose

The pipeline separates **"is this row structurally usable?"** from **"should a
human look at it?"** They are different questions and conflating them is a
common design mistake.

| Gate | Module | Question | Failure goes to |
|------|--------|----------|-----------------|
| 1. Schema validation | `schema.py` | Is the row well-formed (ISO date, non-negative total, known category)? | `rejected[]` |
| 2. Confidence routing | `scoring.py` | Are we confident enough to auto-accept, or does a human need to confirm? | `review_queue/` |

A row can be perfectly valid yet still routed to review because one critical
field came back low-confidence. Conversely, a row OCR read with high confidence
can still be rejected if it's structurally broken. Keeping the gates separate
keeps each one simple and independently testable.

---

## The confidence-threshold router

This is the core of the human-in-the-loop design and the one knob that matters.

```python
# scoring.py
CONFIDENCE_THRESHOLD = 0.85   # rows below this go to review_queue/
```

The aggregate row confidence combines two independent signals:

1. **OCR signal** — the **minimum** per-field confidence across the critical
   financial fields (`date`, `unit_amount_cents`, `total_amount_cents`). We use
   the minimum, not the mean, deliberately: one badly-read amount should sink the
   whole row even if everything else read cleanly. A mean would average that
   single bad field away — hiding exactly the failure we care about.

2. **Consistency signal** — business-logic checks that do **not** depend on OCR
   confidence at all. The main one: does `quantity × unit == total`? A digit
   transposition often reads with *high* OCR confidence yet is still wrong, so
   OCR confidence alone can never catch it. When a consistency check fails we
   apply a penalty that pushes the row below threshold and into review.

The threshold is surfaced in the CLI (`--threshold`) and written into the output
JSON, so it's never hidden inside the code. Tightening it trades more human
review for fewer auto-accepted errors; that tradeoff is a business decision, so
it's a parameter, not a constant baked into logic.

## Quick start

```bash
pip install -r requirements.txt

# Generate the synthetic PDF and run the full pipeline:
python run.py

# Or with a custom threshold / output dir:
python run.py --threshold 0.9 --out output

# Run the tests (includes the page-break edge case):
pytest
```

Running `python run.py` with no arguments produces:

```
=== Extraction summary ===
pages         : 2
threshold     : 0.85
accepted      : 10
review_queue  : 2
rejected      : 0

rows routed to review:
  line 3: conf=0.34 (arithmetic_mismatch)
  line 5: conf=0.55 (low confidence)
```

Outputs land in `output/`:

- `output/extraction_result.json` — full structured result with a summary block.
- `output/review_queue/line_NNN.json` — one file per low-confidence row, so a
  reviewer can claim them individually.

### Real OCR backend (optional)

The default `SyntheticOCREngine` is deterministic and dependency-free, which is
what makes the tests stable. To run real OCR over the generated PDF instead,
install `pytesseract` + `pdf2image` (and the system `tesseract`/`poppler`
binaries) and swap in `TesseractOCREngine` — both engines implement the same
`run(pdf_path) -> OCRResult` interface, so nothing downstream changes.

## Known failure modes

This section is the point. A pipeline like this is only trustworthy if you can
say precisely *how* it breaks. Each item below is a real failure mode of OCR
document extraction; the synthetic OCR engine **deliberately injects** the first
three so the defenses are exercised by the test suite, and the rest are honest
limitations of the current implementation.

### Handled (injected and tested)

1. **Digit transposition in amounts.** OCR can read `$2,530.00` as `$2,350.00`
   with high confidence — the characters are all valid, just reordered. OCR
   confidence will never catch this. The `quantity × unit == total` arithmetic
   check does: row 3 in the sample is seeded with a transposed total, flagged
   `arithmetic_mismatch`, and routed to review.
   *(`scoring.py`; `tests/test_scoring.py::test_arithmetic_mismatch_penalized_even_with_high_ocr_conf`)*

2. **Page-break phantom rows.** When a table spans pages, repeated headers and
   footer fragments at the boundary often OCR into a partial "ghost" row with a
   truncated date (`2024-03-1`) and a stray `$0.00`. Naively parsed, these become
   fake line items that corrupt totals. The parser detects the
   fragment signature (partial-but-not-full date + low line confidence +
   `continued`/`...` markers) and drops them before they reach the schema.
   *(`parser.py::_looks_like_phantom`; `tests/test_parser.py::test_page_break_phantom_row_is_dropped`)*

3. **OCR hallucination — spurious tokens.** OCR sometimes invents a stray glyph
   (`~`, `|`) mid-row, which can shift every column after it and silently
   misalign the money fields. The parser drops very-low-confidence
   non-alphanumeric tokens before column assignment and flags the row
   `dropped_hallucinated_token`. Row 8 in the sample is seeded with one.
   *(`parser.py::_parse_row`; `tests/test_parser.py::test_hallucinated_token_dropped_without_shifting_columns`)*

4. **Faint / smudged fields → low confidence.** A faintly-printed amount comes
   back with low OCR confidence. Because the row score uses the *minimum*
   critical-field confidence, one weak amount sinks the row into review even
   though it's structurally valid. Row 5 in the sample demonstrates this.

### Known limitations (not yet handled — honest gaps)

5. **Column drift on dense tables.** Column assignment is positional/heuristic
   (date first, last two money tokens are unit & total). A document with an extra
   money-shaped column (e.g. a tax column) would mis-assign `total`. A layout-aware
   approach using token bounding boxes would be more robust; the `bbox` field is
   already plumbed through `Token` for exactly this, but the parser doesn't use
   geometry yet.

6. **Multi-line descriptions.** A description that wraps to a second visual line
   is treated as a separate (and likely discarded) OCR line. There's no
   line-merging pass.

7. **Locale / currency assumptions.** Amounts are assumed USD with `$`,
   comma-thousands, dot-decimals. A `1.234,56` (EU) format or a non-`$` currency
   would parse incorrectly. Dates are assumed ISO `YYYY-MM-DD`.

8. **Rotated / skewed scans.** The synthetic engine never produces skew. Real
   Tesseract output on a rotated scan would degrade token grouping; deskew
   preprocessing is out of scope here.

9. **Confidence is not calibrated.** The threshold (0.85) and penalties (0.40 /
   0.50) are reasonable defaults, not values fit to a labeled dataset. In
   production you'd calibrate them against a review-outcome ground truth and
   track the precision/recall tradeoff of the router over time.

## Design notes

- **Money as integer cents.** All amounts are integer cents end-to-end to avoid
  float rounding error in financial data. Formatting to `$` happens only at
  display boundaries.
- **Small dependency surface.** Plain dataclasses instead of pydantic;
  `jsonschema` is used when present but `schema.py` falls back to a built-in
  validator so the pipeline still runs without it.
- **Deterministic by construction.** Injected OCR noise is seeded by row number,
  so every run produces identical output and the tests are stable.
