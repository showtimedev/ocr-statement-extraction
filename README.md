# OCR Statement-of-Loss Extraction

A small document-extraction pipeline. It reads a Statement-of-Loss PDF, runs
OCR, parses line items, validates them against a JSON schema, scores each row's
confidence, and routes low-confidence rows to a human review queue. Output is
structured JSON.

> Synthetic data only. Every byte of data in this repository is generated by
> code. There are no real claims, claimants, policies, dollar amounts, or PII
> anywhere in the project. The sample PDF is produced by `generate_pdf.py` from
> hand-authored fake rows in `sample_data.py`. The repository is a self-contained
> demonstration of the extraction-and-routing pattern. It is not connected to any
> production system or real dataset.

## Problem

Extracting line items from scanned financial documents is easy on clean inputs
and hard on real ones. OCR misreads digits, invents stray characters, and
produces partial rows at page boundaries. A naive "PDF in, JSON out" extractor
passes those errors straight through into structured data, where they are
expensive to catch later.

This project handles the failure cases explicitly. It separates two questions
that are often conflated: "is this row structurally valid?" and "are we
confident enough to accept it without a human?" Rows that fail the first are
rejected. Rows that pass the first but fail the second go to a review queue.

## Stack

- Python 3.11 (developed and tested on 3.11.10).
- `reportlab` for synthetic PDF generation.
- `jsonschema` for line-item schema validation (with a built-in fallback
  validator so the pipeline runs even if the package is absent).
- `pytest` for the test suite.
- Optional real OCR backend: `pytesseract` + `pdf2image` (plus the system
  `tesseract` and `poppler` binaries). The default engine is a deterministic,
  dependency-free synthetic OCR engine.

All Python dependencies are pinned to exact versions in `requirements.txt`.

## Architecture

```
                        +----------------------+
                        |   sample_data.py     |  ground-truth fake rows
                        +----------+-----------+
                                   |
                   +---------------+---------------+
                   v                               v
        +--------------------+          +------------------------+
        |  generate_pdf.py   |          |        ocr.py          |
        |  reportlab -> PDF  |          |  SyntheticOCREngine    |
        +---------+----------+          |  (deterministic noise) |
                  |                     |  TesseractOCREngine    | optional
                  | data/samples/*.pdf  |  (real backend)        |
                  +----------+----------+-----------+------------+
                             |                      | OCRResult
                             v                      v
                        +------------------------------------+
                        |             parser.py              |
                        |  OCR lines -> LineItem rows         |
                        |  - drops page-break phantom rows    |
                        |  - drops hallucinated tokens        |
                        |  - per-field confidences + flags    |
                        +------------------+-----------------+
                                           | list[LineItem]
                                           v
                        +------------------------------------+
                        |   schema.py  (gate 1: structure)    |
                        |   valid? -- no --> rejected[]        |
                        +------------------+-----------------+
                                           | yes
                                           v
                        +------------------------------------+
                        |  scoring.py  (gate 2: confidence)   |
                        |  min(field conf) - consistency pen. |
                        |  >= threshold -> accepted[]          |
                        |  <  threshold -> review_queue/       |
                        +------------------+-----------------+
                                           v
                        +------------------------------------+
                        |  cli.py  ->  output/                |
                        |   extraction_result.json            |
                        |   review_queue/line_NNN.json        |
                        +------------------------------------+
```

Two gates, by design:

| Gate | Module | Question | Failure goes to |
|------|--------|----------|-----------------|
| 1. Schema validation | `schema.py` | Is the row well-formed (ISO date, non-negative total, known category)? | `rejected[]` |
| 2. Confidence routing | `scoring.py` | Are we confident enough to auto-accept, or does a human confirm? | `review_queue/` |

A row can be valid yet still routed to review because a critical field read with
low confidence. A row can read with high confidence yet be rejected because it is
structurally broken. The gates are separate so each stays simple and testable.

## Quick Start

```bash
# From the repository root.
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# Generate the synthetic PDF and run the full pipeline.
python run.py

# Run with a custom threshold and output directory.
python run.py --threshold 0.9 --out output

# Run the test suite (includes the page-break edge case).
pytest
```

Running `python run.py` with no arguments generates the synthetic PDF, runs the
pipeline, and prints a summary:

```

source        : data/samples/statement_of_loss_synthetic.pdf
synthetic     : True
pages         : 2
threshold     : 0.85
accepted      : 10
review_queue  : 2
rejected      : 0

rows routed to review:
  line 3: conf=0.34 (arithmetic_mismatch)
  line 5: conf=0.55 (low confidence)
```

Outputs are written to `output/`:

- `output/extraction_result.json` — full structured result with a summary block.
- `output/review_queue/line_NNN.json` — one file per low-confidence row, so a
  reviewer can claim them individually.

To run real OCR over the generated PDF instead of the synthetic engine, install
`pytesseract` + `pdf2image` (and the system `tesseract` / `poppler` binaries),
then construct the pipeline with `TesseractOCREngine`. Both engines implement the
same `run(pdf_path) -> OCRResult` interface, so nothing downstream changes.

## Design Decisions

- Two separate gates. Structural validity and confidence are different
  questions. Keeping them apart keeps each check small and independently
  testable, and it produces a clear three-way outcome (accepted / review /
  rejected) instead of a single pass-fail.
- Minimum, not mean, for the OCR confidence signal. The row score uses the
  minimum confidence across the critical financial fields (`date`,
  `unit_amount_cents`, `total_amount_cents`). One badly read amount should sink
  the row. A mean would average that single bad field away and hide it.
- A consistency signal independent of OCR confidence. The scorer checks
  `quantity * unit == total`. A digit transposition can read with high OCR
  confidence yet still be wrong, so OCR confidence alone cannot catch it. A
  failed check applies a penalty that pushes the row into review.
- One visible threshold. `CONFIDENCE_THRESHOLD` lives in `scoring.py`, is
  exposed as `--threshold`, and is written into the output JSON. The
  review-versus-accept tradeoff is a business decision, so it is a parameter, not
  a constant buried in logic.
- Money as integer cents end-to-end. Avoids floating-point rounding error in
  financial data. Formatting to dollars happens only at display boundaries.
- Small dependency surface. Plain dataclasses instead of a validation framework.
  `jsonschema` is used when present, with a built-in fallback validator so the
  pipeline still runs without it.
- Deterministic by construction. Injected OCR noise is seeded by row number, so
  every run produces identical output and the tests stay stable.

## Known Limitations & Failure Modes

Handled and tested. The synthetic OCR engine deliberately injects the first three
so the defenses are exercised by the suite.

1. Digit transposition in amounts. OCR can read `$2,530.00` as `$2,350.00` with
   high confidence; the characters are valid, just reordered. OCR confidence will
   not catch this. The `quantity * unit == total` check does. Row 3 in the sample
   is seeded with a transposed total, flagged `arithmetic_mismatch`, and routed
   to review.
   (`scoring.py`; `tests/test_scoring.py::test_arithmetic_mismatch_penalized_even_with_high_ocr_conf`)

2. Page-break phantom rows. When a table spans pages, repeated headers and footer
   fragments at the boundary can OCR into a partial row with a truncated date and
   a stray `$0.00`. Parsed naively, these become fake line items. The parser
   detects the fragment signature (partial-but-not-full date, low line
   confidence, `continued`/`...` markers) and drops them before the schema gate.
   (`parser.py::_looks_like_phantom`; `tests/test_parser.py::test_page_break_phantom_row_is_dropped`)

3. OCR hallucination, spurious tokens. OCR sometimes invents a stray glyph
   mid-row, which can shift every column after it and misalign the money fields.
   The parser drops very-low-confidence non-alphanumeric tokens before column
   assignment and flags the row `dropped_hallucinated_token`. Row 8 in the sample
   is seeded with one.
   (`parser.py::_parse_row`; `tests/test_parser.py::test_hallucinated_token_dropped_without_shifting_columns`)

4. Faint or smudged fields. A faintly printed amount returns low OCR confidence.
   Because the row score uses the minimum critical-field confidence, one weak
   amount sends the row to review even though it is structurally valid. Row 5 in
   the sample demonstrates this.

Known gaps, not yet handled.

5. Column drift on dense tables. Column assignment is positional and heuristic
   (date first; last two money tokens are unit and total). A document with an
   extra money-shaped column (for example a tax column) would mis-assign the
   total. A layout-aware approach using token bounding boxes would be more robust.
   The `bbox` field is already plumbed through `Token` for this, but the parser
   does not use geometry yet.

6. Multi-line descriptions. A description that wraps to a second visual line is
   treated as a separate OCR line and likely discarded. There is no line-merging
   pass.

7. Locale and currency assumptions. Amounts are assumed to be USD with `$`,
   comma thousands separators, and dot decimals. A `1.234,56` European format or a
   non-dollar currency would parse incorrectly. Dates are assumed ISO
   `YYYY-MM-DD`.

8. Rotated or skewed scans. The synthetic engine never produces skew. Real
   Tesseract output on a rotated scan would degrade token grouping. Deskew
   preprocessing is out of scope here.

9. Confidence is not calibrated. The threshold (0.85) and the penalty values are
   reasonable defaults, not numbers fit to a labeled dataset. This repository
   reports no accuracy metric because none has been measured against ground-truth
   review outcomes.

## What I'd Build Next

- Geometry-aware column assignment using the `bbox` data already carried on each
  token, to remove the positional-heuristic fragility in gap 5.
- A line-merging pass for wrapped multi-line descriptions (gap 6).
- A calibration harness: a labeled set of review outcomes, plus a script that
  sweeps the threshold and reports the precision/recall tradeoff so the value can
  be chosen from data rather than picked by hand.
- A second document layout to test that the parser generalizes beyond the single
  synthetic template.
- A thin review UI that reads `review_queue/` and lets a human accept, correct,
  or reject each row, writing the decision back as labeled data for calibration.
